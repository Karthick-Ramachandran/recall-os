# Opinion Packs

## Naming

The user-facing CLI term is:

```txt
preset
```

The architecture term is:

```txt
opinion pack
```

This keeps the CLI familiar while making the internal model explicit.

## Purpose

Opinion packs are optional starting points.

They can generate stack-aware memory docs, workflow guidance, templates, and proposed ADRs for a
specific ecosystem or team style.

Examples:

```bash
persist init --preset nextjs
persist init --preset flutter
persist init --preset startup
```

`persist init --preset flutter` means neutral repository memory plus Flutter-aware guidance and
proposed decisions.

## Implementation Direction

P4 started with preset contracts and minimal built-in content.

The contract is now proven: presets ship, validate, render through the safe write pipeline, and stay
non-authoritative. Rich framework guidance is therefore added under the content standard below.

## Opinionated Preset Content Standard

An opinionated preset must follow this shape so every stack reads consistently:

1. A rich guidance template at `docs/ai/presets/<id>-guidance.md` that:
   - frames every choice as proposed, not accepted;
   - names the stack's real decision forks (for example UI framework, concurrency, persistence,
     dependency injection or data layer, and testing);
   - lists recommended structure, testing, and security considerations.
2. At least four proposed ADRs at `docs/adrs/proposed/ADR-PROPOSED-<id>-<topic>.md`, each with
   Status, Context, Decision, Alternatives, and Consequences sections, all marked `Proposed`.
3. `guidance` entries that summarize the same decision forks for quick reference.

The preset schema enforces neutrality structurally: every preset decision must use
`status: "proposed"`, so richer content cannot silently accept a choice.

## Non-Authority Rule

Opinion packs can suggest.

Opinion packs cannot silently decide.

Any architecture choice generated by an opinion pack must be one of:

- Optional guidance.
- A proposed ADR.
- A clearly marked recommendation awaiting human acceptance.

Opinion packs must not create accepted ADRs by default.

Opinion packs must not silently accept architecture, state management, backend, folder structure,
vendor, or infrastructure choices.

## Valid Opinion Pack Content

An opinion pack may include:

- Ecosystem-specific documentation prompts.
- Suggested module boundaries.
- Suggested testing expectations.
- Suggested security review areas.
- Proposed ADR templates.
- Tool-specific agent instructions.
- Golden output examples.

## Invalid Opinion Pack Content

An opinion pack must not:

- Treat a technology choice as accepted without user approval.
- Override accepted repository decisions.
- Override organization memory.
- Add runtime network behavior in MVP.
- Execute remote templates in MVP.
- Bypass safe write policy.

## Source-Of-Truth Order

Accepted repository decisions and organization memory outrank opinion pack guidance.

If a preset suggests one thing and accepted repository memory says another, the agent must follow
the accepted repository memory or stop and report the conflict.

## Technology Detection

Technology detection may suggest relevant opinion packs or proposed decisions.

Detection must not become accepted repository memory by itself.
