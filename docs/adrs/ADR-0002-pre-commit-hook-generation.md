# ADR-0002: Pre-Commit Hook Generation

## Status

Accepted

## Context

Persist OS validates repository memory through `persist doctor`, but nothing enforces that the gate
runs before changes enter history. Standards and drift checks only help if someone runs them.

A local git pre-commit hook can run `persist doctor` (and a repository's own gates) before each
commit, turning Doctor from an optional command into an enforced gate.

Three constraints shape the design:

- Persist OS Core is architecture-neutral. It must not encode a target repository's toolchain as
  core truth.
- The file write policy is non-destructive and must never escape the project root or silently change
  git behavior.
- A hook is executable and runs on every developer's machine, so it is security-sensitive.

## Decision

`persist init` generates a tracked pre-commit hook at `.persist/hooks/pre-commit`.

- The hook is committed and shared, not written into `.git/hooks`.
- `persist init` proposes the one-time activation command and does not run it:
  `git config core.hooksPath .persist/hooks`.
- The hook runs `persist doctor` first, then each command listed in `preCommitGates` in
  `.persist/config.json`.
- `preCommitGates` is seeded by neutral toolchain detection as proposed, editable values. Detection
  reads `package.json` scripts and a lockfile to suggest commands. If no toolchain is detected, the
  list is empty and the hook runs only `persist doctor`.
- The hook is generated by default during `persist init`, but generation stays non-destructive: an
  existing hook file is skipped unless `--force` is used.

## Alternatives Considered

- Write directly to `.git/hooks/pre-commit`. Rejected: not shared with the team, not reviewable in
  pull requests, and writes into git internals.
- Hardcode auto-detected commands (for example `pnpm test`) directly in the hook. Rejected: it
  encodes a toolchain choice as core truth and conflicts with architecture neutrality. Detection is
  allowed only as a proposed config value the user reviews.
- Run `git config core.hooksPath` automatically during init. Rejected: init must not mutate git
  configuration silently. The command is proposed instead.
- Generate only the hook with no config field. Rejected: gates must be reviewable and editable in
  `.persist/config.json`.

## Consequences

- Doctor and a repository's own gates can be enforced before each commit, closing the gap between
  having a gate and running it.
- Persist OS gains the ability to write an executable file. This is a new file-write capability and
  is constrained to the configured hooks path within the project root.
- `preCommitGates` values are executed as written by the hook. They are user-authored and trusted in
  the same way `package.json` scripts are. The config schema rejects multi-line and control
  characters to reduce accidental injection, and the security model documents the trust boundary.
- Activation remains a deliberate human step (`git config core.hooksPath .persist/hooks`), so init
  never changes git behavior on its own.
- Editing gates in config requires regenerating the hook with `persist init --force`. A future
  `persist hooks sync` command or a Doctor drift check could detect hook-versus-config mismatch.
