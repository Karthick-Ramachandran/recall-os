# PRD: Preset System

## Purpose

Add the core preset system that models optional opinion packs without turning Persist OS Core into
an architecture generator.

P4 proves the preset contract with minimal built-in content.

## Problem

Future `init --preset <name>` flows need a validated way to describe optional stack-aware guidance,
template destinations, and proposed decisions.

Without a preset contract, future commands may treat framework suggestions as accepted repository
memory or bypass safe destination validation.

## Decision

P4 introduces preset schema, validation, registry, and minimal built-in presets.

The CLI term remains:

```txt
preset
```

The architecture term remains:

```txt
opinion pack
```

Presets may suggest. Presets must not silently decide.

## In Scope

- Preset schema.
- Preset validation.
- Preset registry.
- Minimal built-in presets: `generic`, `nextjs`, `ios-swift`, and `flutter`.
- Optional guidance entries.
- Proposed decision stubs with `status: "proposed"`.
- Safe destination validation for template and proposed-decision outputs.
- Duplicate destination detection.

## Non-Goals

- CLI commands.
- `persist init`.
- Template file loading.
- File writes.
- Runtime preset execution.
- Rich framework knowledge bases.
- Technology detection.
- Accepted architecture decisions generated by presets.
- Network, telemetry, MCP runtime, AI API, or cloud behavior.

## Success Criteria

- All built-in presets validate through the public validation path.
- Unsafe destinations are rejected.
- Duplicate normalized destinations are rejected.
- Unknown keys are rejected.
- Accepted decisions are rejected in presets.
- Registry listing is deterministic.
- Registry lookup supports known IDs and clear missing results.
